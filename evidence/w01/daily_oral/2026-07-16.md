## Correct reference answer

LLM Serving 容量规划不能只看字符串长度或请求数，因为模型实际处理的是 token。当前 probe 中，英文输入有 43 个字符但只有 9 个 token，中文输入只有 25 个字符却有 12 个 token，说明字符数与 token 数没有固定比例。请求数也不够，因为同样 100 个请求，可能是一批短 prompt、短输出，也可能是一批长 prompt、长输出，工作量完全不同。

input tokens 主要决定 prefill 阶段需要处理的前缀长度：模型会对整个 prompt 做前向计算，并为后续生成建立各层 KV Cache。output tokens 则通常对应 decode 阶段的自回归迭代次数，每生成一个新 token 都要进行一次新的 decode step。因此容量规划至少要同时记录请求数、input token 数、output token 数以及它们的长度分布。

不过 token 数只是 workload proxy，不等于完整成本。真实延迟和吞吐还会受到模型规模、精度、硬件、并发、batching、调度、KV Cache、prefix reuse、排队和网络开销影响。所以更准确的结论是：以 input/output tokens 作为核心工作量单位，再结合并发和系统配置进行容量规划，而不是只看字符串长度或请求数。
